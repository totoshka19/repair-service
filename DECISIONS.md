# Архитектурные решения

## 1. Next.js App Router + API Routes (монолит вместо отдельного бэкенда)

**Решение:** Единое Next.js-приложение с API Routes в `app/api/`.

**Обоснование:** Для задачи такого масштаба отдельный Express/Fastify-сервер создаёт лишнюю инфраструктуру. App Router позволяет держать серверную и клиентскую части в одном репозитории, деплоить одним контейнером и не тратить время на CORS.

---

## 2. iron-session вместо JWT

**Решение:** Сессия хранится в зашифрованном httpOnly cookie через `iron-session`.

**Обоснование:** JWT требует хранения токена на клиенте (localStorage уязвим к XSS, cookie с JWT всё равно нужен httpOnly). iron-session шифрует данные сессии прямо в cookie с помощью `SESSION_SECRET`, не требует хранилища на сервере, инвалидируется удалением cookie. Для данного проекта это проще и безопаснее, чем JWT + refresh tokens.

---

## 3. Prisma 7 с datasource URL в prisma.config.ts

**Решение:** URL базы данных задаётся в `prisma.config.ts` через `env()`, а не через `env("DATABASE_URL")` в `schema.prisma`.

**Обоснование:** Prisma 7 разделяет конфигурацию схемы и конфигурацию соединения. `prisma.config.ts` — это новый стандарт, который позволяет программно задавать параметры подключения, избегать случайного коммита credentials в схему и поддерживать разные окружения без изменения схемы.

---

## 4. @prisma/adapter-pg (нативный PostgreSQL-адаптер)

**Решение:** PrismaClient создаётся с адаптером `PrismaPg` из `@prisma/adapter-pg`.

**Обоснование:** Prisma 7 переходит на Driver Adapters как основной способ подключения. `PrismaPg` использует пул соединений `pg` напрямую, что даёт контроль над настройками пула и совместимость с edge-окружениями. Стандартный `new PrismaClient()` без адаптера устарел в Prisma 7.

---

## 5. Атомарная защита от гонки в /take

**Решение:** `prisma.request.updateMany({ where: { id, status: 'ASSIGNED', assignedTo: userId } })` — если `count === 0`, возвращаем 409.

**Обоснование:** Наивная реализация (SELECT + проверка + UPDATE) создаёт race condition: два мастера одновременно читают статус `ASSIGNED`, оба проходят проверку и оба обновляют. Атомарный `UPDATE ... WHERE status = 'ASSIGNED' AND assignedTo = userId` выполняется в одной транзакции на уровне PostgreSQL — только один из двух параллельных запросов получит `count = 1`, второй получит `count = 0` и вернёт 409.

---

## 6. bcryptjs вместо bcrypt

**Решение:** Хеширование паролей через `bcryptjs` (pure JavaScript реализация).

**Обоснование:** `bcrypt` требует компиляции нативных C++ модулей (`node-gyp`), что ломается при сборке Docker-образа на Alpine Linux без build tools. `bcryptjs` работает на чистом JS, не требует нативных зависимостей, и для задачи с небольшим числом пользователей разница в производительности несущественна.

---

## 7. proxy.ts вместо middleware.ts

**Решение:** Файл маршрутизации запросов называется `proxy.ts` с экспортом функции `proxy`.

**Обоснование:** Next.js 16 вводит новое соглашение: `middleware` переименован в `proxy`, чтобы избежать путаницы с Express.js middleware. Термин «proxy» точнее описывает поведение — перехват запросов на сетевом уровне перед передачей в приложение. Старое имя `middleware.ts` выдаёт предупреждение при запуске.
